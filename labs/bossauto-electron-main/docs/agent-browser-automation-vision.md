# Agent 驱动的 BOSS 浏览器自动化改造设想

## 1. 背景

当前项目已经验证了基础可行性:

- Electron 应用可以启动。
- Patchright 可以启动 / 附加真实 Chrome。
- 可以通过 CDP 读取 BOSS 页面状态。
- 只读探测已能识别当前 BOSS 页面 URL 与候选人卡片数量。

这说明项目的浏览器自动化底座是可用的。下一步可以讨论是否从"固定流程自动化"升级为"Agent 驱动的受控招聘助手"。

## 2. 核心设想

把成熟 Agent 应用接入多模态大模型,再把 Patchright 封装成浏览器自动化工具,让 Agent 在受控权限内完成 BOSS 招聘页面操作。

目标不是让 Agent 直接乱点网页,而是让它调用一组安全、可审计、面向招聘业务的工具:

```text
scan_candidates
open_candidate
read_resume
close_detail
collect_candidate
greet_candidate
apply_filters
stop_run
```

Agent 负责观察、理解、决策;Patchright 负责执行浏览器动作;业务安全层负责限制可执行范围。

## 3. 分层架构

```text
成熟 Agent 应用(Codex / Trae / Claude Code / Cursor 等)
  ↓
多模态大模型(统一处理文本、截图、页面状态、规则和推理)
  ↓
Agent 工具协议(CLI 或 MCP Server)
  ↓
BOSS 领域工具层(受控动作、结构化状态、权限矩阵)
  ↓
Patchright 浏览器控制层(真实 Chrome、DOM、iframe、截图、点击、复制、滚动)
  ↓
BOSS 直聘网页
```

### Agent 应用层

成熟 Agent 应用负责:

- 规划多步骤任务。
- 调用工具。
- 接收结构化结果。
- 管理对话上下文。
- 向用户解释建议和风险。

### 多模态大模型层

多模态大模型本身已经具备文本理解、视觉理解和推理能力,不应被概念上拆成"视觉模型 + 文本 LLM"。

它的输入可以包含:

- 当前页面截图。
- DOM / Accessibility 摘要。
- 候选人卡片结构化信息。
- 当前 URL。
- 可执行动作列表。
- 岗位筛选规则。
- 已看 / 已联系记录摘要。
- 最近操作日志。
- 异常状态提示。

它的输出应该是结构化决策,例如:

```json
{
  "action": "open_candidate",
  "candidate_id": "abc123",
  "reason": "列表信息通过初筛,需要查看详情"
}
```

### 工具协议层

推荐优先考虑 MCP Server,CLI 作为调试入口。

MCP 更适合:

- 长期浏览器会话。
- 状态保持。
- 结构化工具调用。
- 与 Codex / Trae / Claude Code 等 Agent 集成。

CLI 更适合:

- 本地调试。
- 回归测试。
- 简单脚本化。
- CLI Anything 这类工具进一步包装。

示例 CLI:

```bash
boss-agent observe --json
boss-agent scan-candidates --json
boss-agent open-candidate --id abc123
boss-agent read-resume --json
boss-agent collect --id abc123
boss-agent greet --id abc123
```

## 4. 用户自决模式

系统不应该只有一个"自动 / 不自动"开关,而应允许用户选择自动化程度。

### 模式 1:只读观察

- 只读取页面状态和候选人列表。
- 不打开详情。
- 不写 seen。
- 不收藏。
- 不打招呼。

适合验证页面结构、登录状态、规则效果和 Agent 输出质量。

### 模式 2:人工确认

- Agent 可以扫描、打开详情、读取简历、给建议。
- 收藏 / 打招呼等写操作必须由用户确认。
- 每个候选人保留模型理由和证据。

适合初期上线和用户建立信任。

### 模式 3:半自动

- 低风险动作自动执行,例如记录、跳过、关闭详情。
- 中风险动作可自动执行,例如收藏。
- 高风险动作需要确认,例如打招呼。

适合日常筛选,兼顾效率与安全。

### 模式 4:全自动

- Agent 按规则自动扫描、判断、收藏或打招呼。
- 必须配合批次上限、每日上限、异常暂停、重复保护。
- 用户需要显式开启并理解风险。

适合规则稳定、账号状态稳定、用户明确授权的场景。

## 5. 权限矩阵

```text
动作              只读观察    人工确认    半自动      全自动
读取页面状态        允许        允许        允许        允许
扫描候选卡片        允许        允许        允许        允许
打开候选详情        禁止        允许        允许        允许
读取简历正文        禁止        允许        允许        允许
写 seen 记录        禁止        确认后      允许        允许
跳过候选人          禁止        确认后      允许        允许
收藏候选人          禁止        确认后      允许        允许
打招呼              禁止        确认后      确认后      允许
应用网页筛选        禁止        确认后      允许        允许
批量连续执行        禁止        禁止        限制        限制
```

权限矩阵应由用户配置,并在 UI 中明确展示当前模式会做什么、不会做什么。

## 6. 安全断路器

无论用户选择什么模式,以下情况都应该强制暂停或交给人工:

- 未登录。
- 出现验证码。
- 出现账号安全校验。
- 页面不在 BOSS 推荐页 / 搜索页 / 预期详情页。
- 页面结构异常,无法确认候选人身份。
- 详情正文姓名核对失败。
- 连续读取失败超过阈值。
- 连续点击失败超过阈值。
- 单批动作数超过上限。
- 每日收藏 / 打招呼数超过上限。
- LLM / 多模态模型不可用但当前模式需要模型决策。

安全断路器优先级高于用户模式。即使是全自动模式,也不能越过这些暂停条件。

## 7. 与当前项目的关系

当前项目更接近固定流程自动化:

- `runner.ts` 负责列表扫描、漏斗筛选、打开详情、执行动作。
- `browser.ts` 负责启动 / 附加真实 Chrome。
- `llm.ts` 负责公司、职位、详情判定。
- `actions.ts` 负责收藏 / 打招呼。

Agent 化改造不是推翻现有项目,而是把现有能力拆成更清晰的工具:

```text
现有函数能力 → BOSS 领域工具 → CLI / MCP 暴露 → Agent 调用
```

例如:

- `scanCards` → `scan_candidates`
- `openCardByStableId` → `open_candidate`
- `readDetailText` → `read_resume`
- `collectCurrent` → `collect_candidate`
- `greetCurrent` → `greet_candidate`
- `getStatus` → `observe_page`

## 8. 推荐改造路线

### 阶段 1:只读工具化

- 新增只读 CLI 或内部工具接口。
- 支持 `observe`、`scan-candidates`、`screenshot`。
- 输出稳定 JSON。
- 不做任何点击和写操作。

验收标准:

- 能附加 Chrome。
- 能返回当前 URL、登录状态、候选卡数量。
- 能返回候选卡片结构化列表。

### 阶段 2:受控详情读取

- 增加 `open-candidate`、`read-resume`、`close-detail`。
- 仍不收藏、不打招呼。
- 加入姓名核对、详情读取失败保护。

验收标准:

- 能打开指定候选人详情。
- 能读取简历正文。
- 能确认详情与候选人对应关系。

### 阶段 3:人工确认模式

- 接入 Agent 应用。
- Agent 给出候选人判断和建议动作。
- UI 或 Agent 对话中要求用户确认。

验收标准:

- Agent 能解释为什么建议收藏 / 跳过 / 打招呼。
- 用户确认后才执行写动作。

### 阶段 4:半自动模式

- 开放自动记录、自动跳过、自动收藏。
- 打招呼仍默认确认。
- 增加批次上限和异常暂停。

验收标准:

- 能稳定处理小批次候选人。
- 出错时能暂停并给出原因。

### 阶段 5:全自动模式

- 用户显式授权后启用。
- 所有写动作受权限矩阵、每日限额、异常断路器约束。

验收标准:

- 能在限定数量内完成闭环。
- 所有动作有日志、证据和可回溯理由。

## 9. 关键产品界面

建议 UI 中增加这些区域:

- 模式选择:只读 / 人工确认 / 半自动 / 全自动。
- 动作权限:收藏、打招呼、应用筛选、打开详情。
- 限额配置:每批人数、每日收藏数、每日打招呼数。
- 异常策略:暂停 / 跳过 / 交人工。
- 证据面板:候选卡片、详情正文、截图、模型理由、规则命中。
- 操作日志:每一步动作、模型输出、工具返回、失败原因。

## 10. 关键风险

- Patchright 能降低自动化特征,但不能保证平台永远不风控。
- 多模态模型可能误判页面或候选人。
- 候选人简历涉及隐私,截图和正文要谨慎存储与传输。
- Agent 自主性过强可能误点击、重复联系、错误筛选。
- BOSS 页面结构变化可能导致工具失效。

因此,系统设计重点应是:

- 受控动作。
- 用户自决。
- 证据可见。
- 异常暂停。
- 日志可追溯。

## 11. 待进一步讨论

- 首版应该优先做 CLI 还是 MCP Server。
- 多模态模型是否直接接入现有 Electron,还是通过外部 Agent 应用调用。
- 哪些动作默认允许自动执行,哪些必须确认。
- 是否需要本地保存截图和简历正文,保存多久。
- 现有 `runner.ts` 是保留为传统模式,还是逐步拆成工具集合。
- 是否引入 CLI Anything 帮助生成 / 规范化 Agent CLI。
- Codex、Trae、Claude Code 等 Agent 应用分别如何接入。

# 多项目管理 — 需求与设计方案（v2，已过 codex 评审）

> v2 变更：已吸收 codex 评审的 5 点意见(见 §10)。核心调整:run_plan 从判定标准拆出、迁移改"原样复制"、新增缓存失效与运行中禁切、seen_log 拆"项目级评估 + 全局已联系保护"、明确 boss-filter-guard 是核心改造点。

## 1. 背景与目标

**现状**
- 配置是"单套":一套 BOSS 网页筛选项(`filters.json`)、一套判定标准(`criteria.yaml`,内含 `run_plan`)。
- 筛选项**不区分推荐/搜索**;推荐/搜索的网页筛选有一部分值**写死在 `boss-filter-guard.ts`**,并不读 `filters.json`。
- 全局单例缓存:criteria / filters / seen-log / settings 都是模块级缓存。

**目标**
- 支持**多个"项目"(= 岗位)**,每个项目全套独立:推荐筛选项、搜索筛选项 + 搜索词、判定标准、运行计划。
- 可建多个项目、切换"当前项目";跑的时候用当前项目配置。

## 2. 核心概念：项目（Project）

每个项目独立包含:基本信息、推荐筛选项、搜索筛选项 + 搜索词、判定标准(规则)、运行计划。

**关键调整(codex #1)**:判定标准(规则)与运行计划**分开存**。因为整份判定标准会作为文本喂给 LLM,`run_plan`(跑几个、什么动作)留在里面会污染 LLM 输入,也增加旧文件无损迁移难度。

## 3. 数据结构（示意）

```
Project {
  id, name, notes
  recommendFilters: FilterSettings   // 推荐页筛选项
  searchFilters: FilterSettings      // 搜索页筛选项
  criteria: CriteriaRules            // 只含 hard/company/job_content/decision(喂给 LLM)
  runPlan: RunPlan                   // 运行计划独立(含搜索任务的 keyword)
}
ProjectsConfig { version, activeProjectId, projects[] }
```

## 4. 存储与迁移（无损是硬要求）

- 存储:项目配置放 `userData/projects/`。
- **迁移必须无损(codex #2)**:首次升级、无项目配置时——
  1. 旧 `criteria.yaml` / `filters.json` / `seen_log.json` **原样复制**进"默认项目",**不要 parse 后再 dump**(criteria.yaml 保留原文和注释);
  2. 设为当前项目;**保留旧文件不删**作兜底。
- **读失败不覆盖**:项目配置读/解析失败时保留并报错,绝不当空配置写回(沿用 seen_log 的保护原则)。

## 5. 缓存与并发（codex #3）

- criteria / filters / seen-log / settings 现在都是全局单例缓存。引入"当前项目"后,**切换项目必须统一失效所有这些缓存**,否则会"项目切了、配置还是旧的"(codex 标的最大风险)。
- **运行中(非 idle)禁止切换或删除项目**;只在空闲态允许。

## 6. seen_log 拆两层（codex #5）

- **项目级"评估记录"**(看过 / 判定结果):每个项目独立,不同项目互不影响。
- **全局"已联系保护"**(打招呼 / 收藏):跨项目共享。
- 去重逻辑:项目级记录决定"本项目是否看过";全局保护决定"是否已对此人联系过(任何项目都算)",**防止换个项目又对同一人重复打招呼骚扰**。

## 7. 后端改动

- 新增 `projects.ts`:读写项目配置;`list / getActive / setActive / create / duplicate / rename / archive`。原子写 + 读失败保护。
- `criteria.ts / filters.ts`:改为读写**当前项目**的对应配置;切项目时失效缓存。
- `runner.ts`:跑任务时按 `task.page` 选 `recommendFilters` / `searchFilters`;判定用当前项目 criteria。
- **`boss-filter-guard.ts` 是核心冲突点(codex #4)**:目前筛选值写死、不吃 `FilterSettings`。要把推荐/搜索两套强制逻辑改成**读当前项目对应筛选项**——这是第 4 步的主要工作量。

## 8. 界面（MVP 收敛）

- 顶部**项目选择器**:切换 / 新建 / 复制 / 重命名 / **归档(代替直接删除)**。
- 筛选项区拆成**推荐 / 搜索两块**(搜索块带搜索词)。
- 判定标准、运行计划跟随当前项目;切项目整体刷新。
- **MVP 先做 选择 / 复制 / 重命名 / 归档,不一开始做导入导出**(codex)。

## 9. 分步实施（每步可独立验证、可回滚）

1. **后端数据层**:类型 + `projects/` 存储 + 原样迁移 + 当前项目读写 + 缓存失效。不动界面、不碰抓取。
2. **界面-项目选择器**:切换 / 新建 / 复制 / 重命名 / 归档。
3. **界面-筛选项分区**:推荐 / 搜索两块 + 搜索词。
4. **后端-应用**:按"当前项目 + 页面类型"应用筛选项,把 `boss-filter-guard` 写死值改成读配置。
5. **seen_log 拆分**:项目级评估 + 全局已联系保护。

## 10. codex 评审的 5 点（均已吸收进 v2）

1. run_plan 从判定标准拆出,避免污染 LLM 输入 → §2、§3。
2. 旧文件原样复制迁移,不 parse-dump → §4。
3. 全局单例缓存需随项目切换统一失效,运行中禁切 → §5。
4. boss-filter-guard 硬编码是直接冲突点 → §7。
5. seen_log 拆项目级评估 + 全局已联系保护 → §6。

## 11. 待确认问题 → 采纳 codex 建议的答案

1. **搜索词位置**:放 `run_plan` 的搜索任务(一任务一词)。
2. **存储格式**:criteria 保留 yaml(无损 + 可手改),filters / 项目索引用 json。
3. **run_plan 归属**:项目级独立。
4. **项目粒度**:招聘岗位 / 业务项目。
5. **浏览器 profile / 登录态**:**全局共享**,不随项目切换。
6. **删除项目**:先做"归档",不真删。
7. **运行中切换**:禁止,只在 idle 允许。

## 12. 风险

- **缓存失效不彻底 → 项目间串配置**(codex 标的最大风险),靠 §5 统一失效兜住。
- 迁移非无损 → 丢注释 / 数据,用 §4 原样复制规避。
- boss-filter-guard 改造后要重新核对推荐 / 搜索两套应用是否真生效。
- 与正在改"抓取"的另一个 AI 并行动后端:第 1 步只动数据层(新文件 + 类型),尽量不碰 runner 抓取逻辑;第 4 步动 runner 时再协调。
